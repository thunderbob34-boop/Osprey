## üèÝ Home (Today) ‚Äî 7/10 ‚Üí 9/10

1. ‚úÖ **Weather Coach engine** ‚Äî Open-Meteo forecast + heat lead-up hydration alerts (48h early), best outdoor window today, indoor/shade swap advice, rain/cold guidance. Card on Home; forecast summary feeds the Ozzie daily brief prompt.
2. ‚úÖ **Heat-aware plan actions** ‚Äî WeatherCoachCard shows "Move today's session indoors" whenever `suggestIndoor` is true; marks the session description (Treadmill/Trainer/Indoor) without changing the training stimulus, and the card shows a "Moved indoors ‚úì" state once done.

## üìä Stats (+ Races / Challenges / Calendar) ‚Äî 6/10 ‚Üí 9/10

1. ‚úÖ **Lift analytics** ‚Äî this-week volume (lbs) + top muscle groups worked, an e1RM trend sparkline for the athlete's most-logged lift (Epley formula), and a top-5 PR list by estimated 1RM. All client-side aggregation over existing `exercise_sets` data.
2. ‚úÖ **Per-sport volume breakdown** ‚Äî replaced the single-color "weekly mileage" bar chart with a stacked bar chart of weekly hours by sport (run/bike/swim/lift/cross/race, fixed stack order and per-sport color), plus a legend below showing each sport's total hours (and miles, where distance is meaningful) across the same 6-week window. All client-side aggregation over the same `workout_logs` rows the chart already fetched ‚Äî no new query.
3. ‚úÖ **HealthKit workout import** ‚Äî reads Apple Watch/Garmin (via HealthKit) workouts into `workout_logs` using `getAnchoredWorkouts`, mapping activity names to session types, filtering out OSPREY's own written-back workouts by bundle id, and upserting on a new `(user_id, external_id)` unique index so repeat syncs never duplicate. Runs whenever the user taps "Sync Now" / connects Health in Settings.
4. ‚úÖ **Race predictor across distances for tri** ‚Äî for triathlon-goaled athletes, the RACE PREDICTOR card is now a per-leg predictor: swim/bike/run splits (scaled to their target distance ‚Äî sprint/olympic/half/full) plus a total finish-time estimate, each leg extrapolated (Riegel) from the athlete's own best recorded effort in that discipline. A leg with no logged history shows "Log a swim/bike/run to unlock" rather than a guessed number ‚Äî consistent with Ozzie never inventing data elsewhere. Non-triathletes still see the original run-only 5K/10K/Half/Marathon predictor. ‚öÝÔ∏è Along the way, fixed a pre-existing bug in `fetchPerformanceData` that selected a nonexistent `distance_meters` column instead of `total_distance_km` ‚Äî this silently broke the *entire* Performance Intelligence card (fitness/fatigue/form chart, injury risk, and the run predictor) for every OSPREY+ user, since the query errored on every call. Worth a quick regression check that the fitness/fatigue chart and injury banner render for an existing OSPREY+ account.
5. ‚úÖ **Saved routes with tags** ‚Äî new `saved_routes` table (RLS self-only, GIN-indexed tags) + a Routes screen (reachable from the Stats nav chip row) for adding/removing routes with a name, optional distance, and tags (suggested chips: shaded/indoor/trail/flat/hilly/scenic/loop/track, or custom). `deriveWeatherCoach` now takes the user's saved routes and, on a hot or rainy day, surfaces a matching one (shaded/indoor/trail tags for heat, indoor/covered for rain) as a chip on the WeatherCoachCard ‚Äî and folds it into the same `briefSummary` string already fed to the daily brief, so Ozzie can mention it too without any prompt changes. This is the pragmatic AllTrails substitute.
6. ‚úÖ **Challenges: more types** ‚Äî added `lift_volume` (lbs moved on lift days) and `streak` (longest run of consecutive active-training days in the challenge window) to `challenge_type_enum`. `get_challenge_leaderboard` computes both in dedicated CTEs (lift volume via `exercise_sets`, streak via a gaps-and-islands `ROW_NUMBER()` calc) rather than folding them into the existing aggregate ‚Äî joining `exercise_sets` directly onto that aggregate would have fanned out one row per set and silently corrupted the mileage/workouts/duration counts for every other challenge type. Split across two migrations (enum ADD VALUE, then the function update) since Postgres won't let a new enum value be added and used in the same transaction.

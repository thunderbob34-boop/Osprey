## Findings considered and rejected

- **Rounding identity drift (`tsb` vs `ctl − atl`).** `computeAtlCtlTsb` rounds
  `atl`, `ctl`, and `tsb` independently to 0.1, so `tsb` can differ from
  `displayed_ctl − displayed_atl` by up to 0.1. Cosmetic only, and already
  explicitly acknowledged by an existing test — not worth a plan.
- **Coupled ACWR window (acute ⊂ chronic).** `computeInjuryRisk` averages the
  acute 7 days inside the chronic 28-day window (coupled ACWR). This is a
  legitimate, intentional methodology (Gabbett), not a bug.

## Out of audit scope / follow-ups (not planned)

- **Two TSB *computations*.** `computeAtlCtlTsb` (from `workout_logs`) feeds the
  performance chart, while `daily-summary.ts` reads `row.tsb` from the
  `v_daily_summary` DB view — a second, independent TSB source. Plan 002 makes
  the *labels* consistent but not the underlying *numbers*. Reconciling the SQL
  view against the code-side computation needs database access that was outside
  this audit; flagged for a follow-up.
- **`estimateTss` is intensity- and sport-blind** (`performance.ts`): a flat
  `hours × 50` for any session missing a stored `tss`, so a hard interval and an
  easy jog (or a swim vs a run) contribute identical load. Real accuracy
  limitation, but the fix is a modeling decision (per-sport / per-intensity TSS)
  that needs coaching-domain input — deliberately not turned into a plan here.
  Surfaced to the maintainer as finding #5.

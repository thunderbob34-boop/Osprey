### Known residual gaps (not fixed this run — flagged, not touched)

- **Crossfit `competing` flag is collected but never used.** `crossfit-params.ts` stores it, onboarding and plan-builder both gather it, but `buildCrossfitPrescription` never reads it, despite `docs/coaching/crossfit.md` §1 listing compete-vs-general-fitness as a driver of "programming balance and the peak." Needs a product decision on exactly how it should change the prescription before it's wired in — guessing at the coaching business logic in an unattended pass risked shipping a wrong training effect, not just a wrong display.
- **Hyrox `targetTimeMinutes` is a dead field end-to-end.** Defined, parsed, and stored, but every UI call site hardcodes it to `''`; never actually collected, never read by `buildHyroxPrescription`. Consequence: `hyroxInRaceCarbGPerHour(90)` in `fuel.ts` always assumes a 90-minute race regardless of the athlete's actual target — a sub-75-minute racer gets meaningfully over-prescribed in-race carbs. Fixing this properly means adding the onboarding/preferences UI field and threading it through the envelope, not a one-line patch.
- **Hyrox-specific run zones (`hyroxRunZones` in `calculators/hyrox.ts`) are exported but never wired in.** Because `blueprintSport('hyrox') === 'run'`, hyrox athletes see generic Daniels running zones instead of the blueprint's sec/km compromised-effort offset zones. The compromised-split predictor itself (the main pacing tool) *is* wired correctly — only the display zone model diverges. Needs a design call on whether to swap the display model or keep both.
- **New crossfit/hyrox envelope mirrors widen the pre-existing "client-controlled coaching envelope" trust gap** (flagged in the 2026-07-16 report). The new fields concatenate free-text (`division`, `benchmark.name`, `energySystems[].purpose`, etc.) from the client-supplied envelope into the LLM prompt with no server-side validation — same risk class as before (self-scoped, only affects the caller's own plan), just a larger surface. See feature #3 below for a way to close this as a side effect of a real feature, rather than a standalone hardening pass.
- **StrengthZones card is hidden for a run-primary hybrid athlete with lift data.** The gate is `primary_goal`-only (`lift`/`crossfit`/`hyrox`), so a run-primary athlete with `weekly_lift_days > 0` and populated `oneRepMaxKg` sees no strength zones in the webapp even though mobile generates lift sessions for them. Onboarding currently only collects 1RMs for lift-primary athletes, so this is arguably by-design rather than broken — flagged for a product call, not changed.
- **Momentary two-editor overlap.** If the add form is open and the user clicks an existing session tile, both the add-editor and the session-detail card can render for a single frame before a `useEffect` clears the add state. Cosmetic, self-corrects immediately, not worth restructuring the selection state machine to close.
- **Hyrox division / crossfit "Competing" auto-save on change, while every other field on the same Training Zones card needs an explicit Save.** Both are gated `disabled={saving}` (so there's *some* feedback), but there's no positive "saved" confirmation, unlike the other rows. A full fix (unify the save model, or add save-confirmation to the auto-save controls) was judged more design-opinionated than the mechanical fixes above and was left for a dedicated UX pass.

## Feature recommendations

The 2026-07-16 report already proposed three strong, unbuilt differentiators — Interference Radar, Course IQ, and Fuel Clock — and nothing has shipped against them since. Rather than re-pitch those, these three are chosen to build directly on what *this* audit actually touched: the freshly-shipped plan-editing feature, the cross-sport coaching math verified clean today, and the client-trust gap this audit (and the last one) keeps flagging.

### 1. Load Bridge — a cross-sport fatigue currency
**Pitch.** OSPREY computes load in isolated, per-modality terms — endurance TSS via `computeTrainingLoad`, %1RM-based effort via `strengthWorkingLoads`/`crossfitStrengthLoads` — but nothing unifies them. A crossfitter's heavy engine day and a lifter's max-effort squat session both cost real systemic fatigue that the endurance-only TSB gate never sees, and vice versa for a hybrid athlete's long run the week before a hyrox comp. Load Bridge computes one normalized daily/weekly load number spanning every modality, so the TSB gate and the new webapp Training Zones cards can reason about (and show) one true readiness signal instead of a run-only one. This pushes OSPREY's real moat — one engine spanning strength, endurance, and fuel — somewhere no competitor can follow: Whoop is device-only, TrainingPeaks' TSS is endurance-only, Hevy doesn't touch cardio at all.
**MVP.** A pure function over a week's `workout_logs` (tss for endurance, a %1RM×volume-derived load number for strength/crossfit/hyrox, reusing the already-verified `strengthWorkingLoads`/`crossfitStrengthLoads` outputs) returning one normalized 0–100 daily score. Feed it into `performance.ts`'s existing TSB gate *alongside* (not replacing) tss-only ATL/CTL, and surface it as a single "Today's load" number on the webapp zones card and mobile home screen.
**Effort:** M (3–4 weeks) — every input already exists and was verified correct in this audit; the new work is the strength-load-to-TSS-equivalent conversion formula itself, which needs a defensible calibration (likely off RPE/duration/%1RM) rather than a guess.
**Risk:** A wrong equivalence formula could misdirect the TSB gate. Ship as an additive, clearly-labeled signal first; don't let it silently touch plan-generation guardrails until it's been validated against real athlete data.

### 2. Plan Simulator — a load-aware "what-if" preview on plan editing
**Pitch.** The webapp's brand-new plan-editing feature (this audit's main subject) lets an athlete move or edit any session with zero visibility into the downstream consequence — moving Thursday's interval run onto the day after a heavy lift is exactly the interference collision OSPREY's own coaching philosophy warns against, but the editor has no idea. Plan Simulator adds a live "if you save this" preview inline in `SessionEditor`, before the athlete commits: a projected load delta and whether the move creates a hard-on-hard or heavy-lower-then-threshold-run collision.
**MVP.** A pure preview function taking the current week's sessions plus the pending edit, returning a short flag list ("Wed is now back-to-back with Tue's heavy squat day") rendered as a warning strip between the fields and the Save button — advisory only, no auto-blocking, matching the app's existing "Ozzie explains, athlete decides" tone. Can ship with a light rules-only collision check before any dependency on Load Bridge.
**Effort:** S–M (2–3 weeks) — reuses the just-shipped `SessionEditor`/`session-edit.ts` scaffolding directly, all of which was read line-by-line in this audit.
**Risk:** Low — read-only and advisory; can ship standalone ahead of Load Bridge using simpler per-sport session flags as the first-pass signal.

### 3. Explain My Plan — guardrail transparency (and a real security win)
**Pitch.** This audit and the last one keep flagging the same structural gap: the coaching envelope's guardrails (TSB gate, %1RM caps, polarization-before-clamp ordering) are real and correctly implemented — verified today — but entirely invisible to the athlete, and, because the envelope is client-supplied, entirely untrusted server-side. Explain My Plan turns every generated session into a tappable "why" ("Today's interval volume is capped because your TSB is -18, below the -15 hard-session threshold"), sourced from the actual guardrail evaluation. Building it requires re-deriving that decision server-side from trusted inputs instead of trusting the client's envelope — closing the widened prompt-injection/trust gap flagged above as a side effect, not a separate project. No competitor's black-box plan explains its own math today.
**MVP.** Log which guardrail(s) fired per session in `ozzie-generate-plan` (a new `ozzie_guardrail_reason` field, or a structured pre-step feeding the existing `ozzie_notes` prompt) and render it as an expandable "Why this?" on the session detail view in both mobile and webapp.
**Effort:** L (4–5 weeks) — the logging half is straightforward (the functions already compute these values); making the envelope server-derived instead of client-trusted is a separate, higher-risk hardening project touching `ozzie-generate-plan`'s core contract, and needs care against the envelope byte-identical guarantee this audit just verified.
**Risk:** Scope creep. Sequence it: ship "log + display what already happens" first (low risk, immediate visibility win), treat "stop trusting the client envelope" as a follow-on hardening pass.

**Recommended first bet:** Plan Simulator — smallest effort, ships standalone, and directly extends work this exact audit already reviewed and hardened.

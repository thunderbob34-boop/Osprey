## Fixes implemented

### Security
1. **IDOR in friend-graph RPCs** (`supabase/migrations/20260710000033_fix_friend_rpc_idor.sql`) — `get_my_friends`, `get_friends_at_race`, and `get_pending_friend_requests` were `SECURITY DEFINER` functions filtered on a **client-supplied** `p_user_id` instead of `auth.uid()`. Any authenticated user could pass an arbitrary user's UUID (obtainable from `search_user_by_email` or the leaderboard) and read that user's full friend list and incoming friend requests — a social-graph enumeration. Fixed by dropping the parameter and keying off `auth.uid()`. Also added a race-ownership check to `get_race_partners`, which previously returned partner names for any `race_id` with no ownership check. Updated the three client call sites (`src/services/challenges.ts`, `friends.ts`, `racePartners.ts`) to match.
2. **Two edge functions had no auth check** (`supabase/functions/ozzie-race-briefing`, `ozzie-race-retro`) — every other edge function validates the caller via `supabase.auth.getUser(token)`; these two didn't, so anyone holding the shipped app's public anon key (itself a valid JWT) could call them unauthenticated — free GPT-4o-mini proxy abuse, no per-user cost gating. Added the same `getUser()` gate the other six functions use, plus a 15s `AbortSignal.timeout` on the OpenAI call (neither function had a timeout before).

### Data integrity
3. **`coach_memory` writes were silently failing on every call** (`supabase/migrations/20260710000034_fix_coach_memory_conflict_indexes.sql`) — the highest-value functional fix in this audit. Both upserts (`recordPrMemory` in `workouts.ts`, `recordRaceResult` in `races.ts`) target `onConflict` columns backed only by **partial** unique indexes (`WHERE workout_id IS NOT NULL` / `WHERE race_id IS NOT NULL`). PostgREST's `onConflict` only ever passes column names, never a `WHERE` predicate, so Postgres can't infer a partial index for `ON CONFLICT` and every insert threw `42P10`, swallowed by a `console.error`. **Net effect: the `coach_memory` table has been silently empty this whole time**, and the daily-brief "remember your PR / your last race" callback loop it feeds has never fired. Fixed by making both indexes non-partial (btree uniqueness already treats NULLs as distinct, so this changes nothing else).
4. **Migration replay was broken from a clean database** (`supabase/migrations/20260628000001_initial_schema.sql`, `20260628000004_fix_remaining_rls.sql`) — migration 001 created a `saved_routes` table with one schema (`surface`/`gpx_url`/`start_lat`/`start_lon`/`is_public`/`deleted_at`), and migration 023 (unmodified) later creates the *actual* table the app uses (`tags`/`distance_km`/`notes`) with no `IF NOT EXISTS` — so replaying migrations 001→032 on a fresh database fails outright at 023 with "relation already exists." Confirmed via grep that nothing in migrations 001-022 references any of 001's `saved_routes` columns (they're dead), so removed that duplicate block from 001 and its matching (now-orphaned) RLS policies from 004. **Action needed from you:** I can't check the live database's actual `saved_routes` schema from here — if your production table doesn't have `tags`/`notes` columns, the Saved Routes feature is currently broken for real users, not just for fresh installs. Worth a quick check in the Supabase SQL editor: `select column_name from information_schema.columns where table_name='saved_routes'`. Also note editing an already-applied historical migration file may require `supabase migration repair --status applied 20260628000001 20260628000004` before your next `db push` (doesn't rerun the migration, just reconciles the CLI's tracking).
5. **FK RESTRICT could block plan deletion** (`supabase/migrations/20260710000035_session_fk_on_delete_set_null.sql`) — `workout_logs.session_id`, `plan_adjustments.session_id`/`plan_id` had no `ON DELETE` action (defaults to RESTRICT). Once any workout is logged against a session, deleting that session's parent plan fails with an FK violation instead of cascading. `ozzie-generate-plan` already works around this by manually nulling `session_id` before deleting; this migration makes it safe by default for any other delete path. Both columns are nullable, so `SET NULL` is valid.

### Feature bugs
6. **Paywall showed the annual price as a monthly price** (`app/paywall.tsx`) — the subscribe button hardcoded `/mo` onto every price. Since the annual package is pre-selected by default, the CTA read **"Start for $59.99/mo"** (real price: $59.99/**yr**) — a $720/yr-equivalent misread, and a concrete App Store review risk (accurate-pricing guideline). Added a `periodSuffix()`/`periodSpokenSuffix()` helper keyed on `packageType` so the button and VoiceOver label both show the correct period for whichever plan is selected.
7. **HealthKit writes silently dropped distance and calories** (`src/services/healthkit.ts`) — `saveWorkout` nested `distanceMeters`/`calories` under a `metadata` key; the native module (`RCTAppleHealthKit+Methods_Workout.m`) reads top-level `distance`/`distanceUnit`/`energyBurned`/`energyBurnedUnit` and hardcodes `metadata: nil`. Every synced workout landed in Apple Health with correct type/duration but **0 distance, 0 calories**, with no error (the callback still reports success). Moved the fields to top-level.
8. **Swim/bike/rowing/hyrox workouts were never written to Apple Health at all** (same file) — the session-type → HealthKit-activity map only had `run`/`lift`/`cross`; those four other logged workout types silently produced no HealthKit write (swallowed by a `.catch`). Added `swim → Swimming`, `bike → Cycling`, `rowing → Rowing`, `hyrox → HighIntensityIntervalTraining`.
9. **Password reset only worked on a cold app launch** (`app/reset-password.tsx`) — recovery tokens were parsed only from `Linking.getInitialURL()`. If the app was already running/backgrounded when the user tapped the reset-email link (the common case — request reset, switch to email, tap link), the app warm-starts instead of cold-launching, `getInitialURL()` returns nothing useful, and the user sees "invalid or expired link" for a perfectly valid one. Added a `Linking.addEventListener('url', …)` subscription alongside the existing cold-start check.
10. **PR badge over-fired on a brand-new exercise's first-ever log** (`src/services/workouts.ts`) — `detectSetPr` returned `true` (celebrate as PR) whenever there was no prior history at all, while the *live* mid-workout check in `lift.tsx` requires `previousBest > 0` and never celebrates a first log. Same session could show no PR haptic mid-workout but "New PR!" on the recap. Aligned the recap check to the live one (no history ≠ PR).
11. **Friends list had no error state** (`app/friends.tsx`) — a failed fetch fell through to the "no friends yet" empty-state copy, indistinguishable from actually having zero friends (every other list screen in the app has an explicit error branch). The hook already exposed `error`; wired it up.

All fixes verified against the actual native/library source where relevant (react-native-health's Objective-C implementation, RevenueCat's `PurchasesPackage.packageType` enum) rather than assumed.

This is a business-model question, not a bug I should guess at: plan generation is core to onboarding, so gating it could hurt signup conversion; leaving it ungated while advertising it as paid is a paywall-copy misrepresentation. Needs a decision on intended behavior before either the copy or the code changes.
- **"Try it free" is promised in the App Store description/promo text, but no trial exists in the paywall or RevenueCat config** (per `docs/TODO.md`, both products are plain paid). Marketing-copy vs. product mismatch — needs either a configured intro offer or copy edit.
- **GPS route map is captured but never shown again after a run ends** — `RunMap` only renders during the live run; the saved track is read back only to compute mile splits, never rendered as a map on the recap or elsewhere, despite both the QA checklist and the store description promising a "route map." Real feature gap, but a genuine UI feature to build (recap map view), not a one-line fix — sizing it is in the recommendations below as a natural extension.
- `get_activity_feed` RPC referenced in `src/services/activity.ts` (with a client-supplied `p_user_id`, the same pattern just fixed elsewhere) **does not actually exist in any migration** — every call fails and silently falls back to `fetchActivityFeedSimple` (a plain RLS-scoped query, which is safe). Not a live IDOR since the function was never created, but it's dead, permanently-failing code worth cleaning up or finishing.
- Coaching-engine math itself came back clean: all 10 sport calculators (running/cycling/swimming/rowing/ultra/hyrox/triathlon/powerlifting/crossfit) were checked formula-by-formula against `docs/coaching/` and match exactly. ## Feature recommendations

Three proposals chosen to differentiate OSPREY from single-sport trackers (Strava, Garmin) and static-plan platforms (TrainingPeaks) by leaning into what's already architecturally unique here: the 4-input Expert Coach engine, `coach_memory` (now actually working), and the hybrid strength+endurance identity in `docs/coaching/_index.md`.

### 1. Proactive ACWR-based plan rebalancing
**Pitch:** Instead of reacting to today's TSB dipping negative, catch a week climbing toward the 1.3–1.5 ACWR danger zone a few days early and rebalance the plan before overtraining symptoms show up — most competitors surface injury-risk *scores* (Whoop, Garmin) but don't act on them.
**Scope:** `computeInjuryRisk` and `computeAtlCtlTsb` (`src/services/performance.ts`) already compute the trend data this needs; `usePlanDeload.ts` already has a same-day threshold check. Extend it to look at the ACWR trend over the trailing 3-4 days, and when it's climbing toward the danger band, trigger a proactive weekly rebalance using the existing `compressTodaySession`/`swapTodaySession` mutation pattern instead of a single-day suggestion.
**Effort:** Medium (1-2 weeks) — mostly wiring existing computed data into a new trigger path; no new calculators or schema.
**Integration points:** `src/hooks/usePlanDeload.ts`, `src/services/performance.ts`, existing session-swap mutations in `src/services/workouts.ts`.

### 2. Interference-aware hybrid session sequencing
**Pitch:** OSPREY's stated identity is "look like a bodybuilder, function like an athlete" — a hybrid athlete's single biggest silent plan-quality problem is *interference* (heavy legs the day after a long run blunts both adaptations; Z2 endurance done after a max-effort lift session is compromised cardio). No competitor in this space (Hevy, TrainingPeaks, Whoop) manages this automatically for the hybrid case — it's exactly the niche OSPREY's brand already claims.
**Scope:** Add an interference-check pass over `training_sessions` at plan-generation time (and at drag-to-reschedule time) using the zone/intensity metadata `docs/coaching/_index.md` and the per-sport files already define (e.g., don't sequence a heavy-leg day immediately after/before a long run or interval session without a minimum recovery gap; front-load endurance before strength on shared days rather than after). Surface a plain-language explanation when the coach reorders something ("moved your long run before Thursday's leg day so neither one is compromised").
**Effort:** Medium-large (2-3 weeks) — the constraint logic is new, but it plugs into the existing `ozzie-generate-plan` session-assignment step and the existing plan-adjustment/`ozzie_reason` explanation pattern already used elsewhere (`plan_adjustments.ozzie_reason`).
**Integration points:** `supabase/functions/ozzie-generate-plan`, `src/services/plan.ts` (`computeRacePhase` and periodization logic live here), `docs/coaching/_index.md` zone definitions.

### 3. Post-run route replay + live pacing copilot
**Pitch:** Two features sharing infrastructure. (a) Show the GPS route that was just run — currently captured and immediately write-only (see finding above), a real gap versus what's promised. (b) Extend the existing on-screen cue banner (`useCueBanner`, added when Ozzie voice was disabled) into a *dynamic* pacing coach that adjusts target pace mid-run based on live conditions (elevation profile from the route, current temperature/humidity from `weather-coach.ts`) rather than a single static goal pace — something Strava/Garmin don't do for amateur athletes without a paid human coach.
**Scope:** (a) is small — render `RunMap` (already built) on `app/workout/recap.tsx` using the GPS track already persisted and already read back for splits. (b) is larger — feed `weather-coach.ts`'s existing forecast fetch and the route's elevation samples into the pace-band logic in `services/calculators/running.ts` (and equivalents) to produce a segment-by-segment adjusted target, surfaced via the existing `useCueBanner`.
**Effort:** Small for (a) (~2-3 days, reuses an existing component); medium for (b) (~1-2 weeks, new pacing-adjustment logic but reuses existing weather/GPS/cue-banner plumbing).
**Integration points:** `app/workout/recap.tsx`, `src/components/RunMap` (+ `.web` variant), `src/services/weather-coach.ts`, `src/services/calculators/running.ts`, `src/hooks/useCueBanner.ts`.
